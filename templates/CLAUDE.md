# CLAUDE.md — Triple Loop 自律開発システム グローバル設定

> **配置場所**: このファイルを `~/.claude/CLAUDE.md` に配置することで、  
> すべてのプロジェクトで有効なグローバル動作ルールとして機能します。  
> プロジェクト固有のルールは各プロジェクトルートの `AGENTS.md` に記述してください。

---

## このファイルの役割

`~/.claude/CLAUDE.md` は Claude Code が起動するたびに最初に読み込む  
**グローバル設定ファイル**です。以下を定義します:

1. **Triple Loop システム** — Monitor / Build / Verify の3層自律実行ルール
2. **自律実行モード** — 停止条件・エスカレーション条件
3. **Agent Teams オーケストレーション** — 専門エージェントの役割分担
4. **Git 操作ルール** — コミット・ブランチ・PR の自動操作基準
5. **コスト最適化** — モデル選択・バッチ処理・コンテキスト管理
6. **エラーハンドリング** — エラー種別と対応アクションの定義

## セッション開始プロトコル

Claude Code がセッションを開始するとき、**必ず以下の順序**で実行してください:

```
Step 1: git status && git log --oneline -10
Step 2: TASKS.md が存在する場合 → Pending タスクを読む
Step 3: AGENTS.md が存在する場合 → コーディング規約を読む
Step 4: .loop-monitor-prompt.md が存在する場合 → Monitor Loop 設定を読む
Step 5: 作業内容を1文で宣言してから実装を開始する
```

**宣言フォーマット**:
```
▶ 開始: [タスク名] を [ブランチ名] で実装します。
  対象ファイル: [主要ファイル3件以内]
  完了基準: [テスト通過 / lint クリア / カバレッジ X%]
```

## Triple Loop システム — 全体アーキテクチャ

```
┌─────────────────────────────────────────────────────────────────┐
│                    Triple Loop 15H システム                       │
│                                                                   │
│  ┌─────────────────┐                                             │
│  │  Monitor Loop   │ ← GitHub Issues / PRs / CI ステータス監視   │
│  │  (15分サイクル) │   TASKS.md 優先度更新                        │
│  └────────┬────────┘                                             │
│           │ タスクキュー供給                                       │
│           ▼                                                       │
│  ┌─────────────────┐                                             │
│  │   Build Loop    │ ← TASKS.md Pending → 実装 → 品質ゲート       │
│  │  (タスク毎)     │   失敗時: 最大3回リトライ → エスカレーション    │
│  └────────┬────────┘                                             │
│           │ 実装完了                                              │
│           ▼                                                       │
│  ┌─────────────────┐                                             │
│  │  Verify Loop   │ ← コードレビュー / セキュリティ / PR 作成      │
│  │  (PR毎)        │   全チェック通過 → マージ承認待ち               │
│  └─────────────────┘                                             │
└─────────────────────────────────────────────────────────────────┘
```

## Monitor Loop — 実行ルール

### トリガー条件

| トリガー | 頻度 | 実行内容 |
|---------|------|---------|
| セッション開始時 | 毎回 | 全 Pending タスクのスキャン |
| タスク完了後 | タスクごと | TASKS.md の Done 更新 + 次タスク選択 |
| Build Loop 失敗後 | 失敗ごと | エラー分析 + タスク優先度再評価 |
| 定期ヘルスチェック | 15分おき | CI ステータス / Issue 変更検出 |

### TASKS.md 優先度スコアリング

```
優先度スコア = (severity_weight × 3) + (age_days × 0.5) + (blocker_count × 2)

severity_weight:
  critical: 10  |  high: 7  |  medium: 4  |  low: 1

例: critical + 5日経過 + 2件ブロック = 10×3 + 5×0.5 + 2×2 = 36.5点
```

### Monitor Loop 出力先

```
TASKS.md       → タスクステータス更新（Pending / In Progress / Done / Blocked）
.loop-alert.md → エスカレーション通知（人間への判断要請）
SESSION-SUMMARY.md → セッション進捗サマリー
```

## Build Loop — 実行ルール

### タスク着手条件

Build Loop が新しいタスクに着手するには以下がすべて満たされる必要があります:

```
✅ TASKS.md に Pending タスクが存在する
✅ 前のタスクが Done または Blocked に移行済み
✅ 依存タスクがすべて Done
✅ ブランチが main/master から最新状態
✅ 既存のビルドが PASS している
```

### 品質ゲート定義（全プロジェクト共通）

```
BUILD GATE（ビルド通過）:
  - コンパイルエラー: 0件
  - ビルド時間: ベースラインの +30% 以内

TEST GATE（テスト通過）:
  - 全テスト: PASS
  - カバレッジ: プロジェクト設定値以上（デフォルト ≥ 80%）
  - テスト時間: ベースラインの +20% 以内

LINT GATE（静的解析通過）:
  - エラー: 0件
  - 警告: 新規追加 0件（既存警告はフリーズ）

TYPE GATE（型チェック通過）:
  - 型エラー: 0件
  - any 型の新規追加: 0件

SECURITY GATE（セキュリティ通過）:
  - Critical / High 脆弱性: 0件
  - シークレットのハードコード: 0件
```

### Build Loop 実行サイクル

```
1. タスク選択（Monitor Loop のスコア上位）
2. コンテキスト収集（関連ファイル特定）
3. 実装（プロンプト: build-loop-prompts.md）
4. 品質ゲート実行
   PASS → コミット → Verify Loop へ
   FAIL → 原因分析 → リトライ（最大3回）
5. 3回失敗 → エスカレーション（.loop-alert.md 生成）
```

### リトライ間隔とバックオフ

```
1回目失敗: 即座にリトライ（同じアプローチで修正）
2回目失敗: 別アプローチで試行（リファクタリング検討）
3回目失敗: エスカレーション + タスクを Blocked に移動

リトライ前に必ず:
  - 前回の失敗原因をコメントに記録する
  - 変更内容を git stash で一時退避してから再実装する
```

## Verify Loop — 実行ルール

### Verify Loop トリガー

Build Loop がすべての品質ゲートを通過したとき、自動的に Verify Loop が起動します。

### 検証チェックリスト

```
Stage 1: コードレビュー（/review コマンド）
  [ ] 変更スコープが要件に限定されている
  [ ] マジックナンバー・ハードコード文字列がない
  [ ] エラーハンドリングが適切
  [ ] ログレベルが適切（debug/info/warn/error の使い分け）
  [ ] コメントが Why を説明している（What は不要）

Stage 2: セキュリティレビュー
  [ ] インジェクション脆弱性（SQL/Command/LDAP）
  [ ] 認証・認可の抜け漏れ
  [ ] 機密情報のハードコード
  [ ] CORS / CSP 設定の確認
  [ ] 入力値のバリデーション

Stage 3: パフォーマンスレビュー
  [ ] N+1 クエリの有無
  [ ] 大量データ処理での効率（ページネーション・ストリーミング）
  [ ] キャッシュの適切な使用

Stage 4: PR 作成（全 Stage PASS 後）
  [ ] PR 説明文の自動生成（verify-loop-prompts.md プロンプト5）
  [ ] レビュアーのアサイン
  [ ] ラベルの付与
```

### Verify Loop 判定基準

| 結果 | 条件 | アクション |
|-----|------|----------|
| ✅ 承認 | 全 Stage PASS | PR をマージ承認待ち状態にする |
| ⚠️ 要修正 | Medium 以下の問題 | 修正後に Verify Loop 再実行 |
| 🔴 要確認 | High / Critical | 人間エンジニアへのレビュー要請 |
| ⛔ 拒否 | Critical 脆弱性 | PR をドラフトに戻し即座に停止 |

## 自律実行モード — 停止条件とエスカレーション

### ✅ 自律実行を続行してよいケース

```
# コード品質の問題（自動修正可能）
- テスト失敗（原因が特定できる場合）
- lint / typecheck エラー
- コンパイルエラー
- カバレッジ不足（テスト追加で解消可能）
- コードスタイル違反
- リファクタリングによる軽微な変更

# 技術的判断が明確な場合
- ドキュメント更新
- 依存パッケージのパッチバージョンアップデート
- テストの追加・修正（既存テストを壊さない範囲）
```

### 🛑 停止して人間に報告すべきケース

```
SECURITY（最優先停止）:
  - シークレット・APIキーのコードへの混入を検出
  - 認証バイパスにつながる可能性のある変更
  - Critical / High のセキュリティ脆弱性の発見
  - 個人情報・機密データの意図しない露出

DESTRUCTIVE（データ破壊リスク）:
  - DROP TABLE / DROP DATABASE / TRUNCATE を含む変更
  - 本番データへの直接書き込み・削除
  - マイグレーションのロールバック不可変更
  - 大量のファイル削除（50ファイル超）

ARCHITECTURE（設計判断が必要）:
  - パブリック API の破壊的変更（後方互換性の喪失）
  - データベーススキーマの大規模変更
  - サービス分割・統合などのアーキテクチャ変更
  - 外部サービスへの新規依存追加

REPEATED FAILURE（技術的限界）:
  - 同一問題での3回連続失敗
  - テスト失敗の根本原因が特定できない
  - ビルドエラーの修正がループしている
  - 修正が別の問題を引き起こすカスケード障害
```

### 停止時の報告フォーマット

```markdown
# 🚨 停止レポート: {timestamp}

## 停止理由
**カテゴリ**: SECURITY | DESTRUCTIVE | ARCHITECTURE | REPEATED_FAILURE
**概要**: [1行で状況を説明]

## 現在の状態
- **作業中タスク**: {task_title}
- **完了した作業**: {completed_steps}
- **未完了の作業**: {remaining_steps}
- **変更ファイル数**: {changed_files_count}

## 問題の詳細
```
{error_message_or_concern_detail}
```

## 試みた対応
1. {attempted_fix_1}: {result_1}
2. {attempted_fix_2}: {result_2}
3. {attempted_fix_3}: {result_3}

## 人間に必要な判断
- [ ] {decision_1}: 選択肢A / 選択肢B
- [ ] {decision_2}: 承認 / 却下

## 再開手順
1. `git stash show` で保留中の変更を確認
2. 判断後: `{resume_command}` で再開
3. 破棄する場合: `git checkout .` でリセット
```

## Agent Teams オーケストレーション

### エージェント役割定義

| エージェント ID | 専門領域 | 起動タイミング |
|--------------|---------|-------------|
| `@monitor-agent` | GitHub 監視・タスク管理 | セッション開始時・定期実行 |
| `@backend-agent` | API・DB・認証・ビジネスロジック | バックエンドタスク |
| `@frontend-agent` | UI・UX・コンポーネント・CSS | フロントエンドタスク |
| `@test-writer` | テスト設計・生成・カバレッジ改善 | テスト不足時・TDD |
| `@security-agent` | 脆弱性検出・セキュリティレビュー | Verify Loop Stage 2 |
| `@docs-agent` | README・API仕様・変更履歴生成 | PR 作成前 |
| `@refactor-agent` | コード品質・複雑度改善 | リファクタリングタスク |

### 並列実行ルール（/fleet コマンド）

```
# 並列実行を許可する条件
✅ 対象ファイルに重複がない（書き込み競合なし）
✅ 各タスクが独立して品質ゲートをパスできる
✅ 依存関係がない（タスク A が タスク B の出力を必要としない）
✅ 合計コスト見積もりが予算内

# 並列実行の例
/fleet @backend-agent でタスクA, @frontend-agent でタスクB を並列実行
→ 両方の品質ゲートを待機してから merge

# 並列実行禁止ケース
❌ 同一ファイルへの並列書き込み
❌ DB スキーマ変更と API 変更の同時実行
❌ 同一テストスイートの並列修正
```

### エージェント間通信プロトコル

```
エージェントは作業結果を以下のファイルに記録して引き渡します:

.agent-handoff/{agent_id}-output.json
{
  "agent_id": "backend-agent",
  "task_id": "TASK-042",
  "status": "completed|failed|escalated",
  "artifacts": ["src/api/...", "src/services/..."],
  "test_results": {"passed": 24, "failed": 0},
  "notes_for_next_agent": "..."
}
```

## Git 操作ルール

### ブランチ命名規則

```
機能追加: feat/{task-id}-{kebab-case-description}
バグ修正: fix/{task-id}-{kebab-case-description}
リファクタリング: refactor/{task-id}-{kebab-case-description}
緊急修正: hotfix/{issue-id}-{kebab-case-description}
ドキュメント: docs/{description}

例:
  feat/TASK-042-add-stripe-payment
  fix/TASK-055-duplicate-charge-prevention
  hotfix/SEC-001-auth-bypass-patch
```

### コミット規約（Conventional Commits 準拠）

```
<type>(<scope>): <日本語または英語の説明> [#issue番号]

type:
  feat     - 新機能（MINOR バージョンアップ相当）
  fix      - バグ修正（PATCH バージョンアップ相当）
  docs     - ドキュメントのみの変更
  refactor - 動作変更なしのコード改善
  test     - テストの追加・修正
  chore    - ビルド・CI・設定変更
  security - セキュリティ修正（PATCH 相当、即時リリース優先）
  perf     - パフォーマンス改善
  revert   - コミットの取り消し

scope（任意）: api | db | auth | ui | config | deps | docs

例（良い例）:
  feat(api): POST /payments エンドポイントを追加 #42
  fix(auth): JWTリフレッシュトークンの有効期限バグを修正 #67
  security(deps): lodash を 4.17.21 に更新（CVE-2021-23337対応）

例（悪い例）:
  ❌ "fix bug"
  ❌ "update code"
  ❌ "WIP"（ドラフト PR は除く）
```

### コミットの粒度ルール

```
✅ 推奨コミット粒度:
  - 1コミット = 1つの論理的変更
  - 品質ゲートがパスした状態でのみコミット
  - ファイル数: 目安 1〜10 ファイル / コミット

✅ コミット前チェックリスト:
  [ ] git diff --staged でステージ内容を確認
  [ ] 不要なデバッグコード・console.log がない
  [ ] .env や シークレットが含まれていない
  [ ] コミットメッセージが規約に沿っている

❌ コミット禁止ケース:
  - テストが失敗している状態
  - ビルドエラーがある状態
  - シークレットが含まれている状態
  - main/master への直接コミット
```

### PR 作成ルール

```
PR 作成タイミング:
  - Verify Loop の全チェックが PASS した後
  - 少なくとも 1 件のテストが追加・更新されている
  - CHANGELOG.md が更新されている（機能追加・バグ修正の場合）

PR テンプレート準拠:
  - .github/pull_request_template.md が存在する場合は必ず使用
  - 存在しない場合は verify-loop-prompts.md プロンプト5 を使用

PR サイズ基準:
  S サイズ: 変更行数 < 100行（レビュー負荷: 低）
  M サイズ: 変更行数 100〜400行（レビュー負荷: 中）
  L サイズ: 変更行数 400〜1000行（レビュー負荷: 高 → 分割推奨）
  XL サイズ: 変更行数 > 1000行（原則として分割必須）
```

## コスト最適化設定

### モデル選択ガイドライン

```
claude-3-5-haiku（低コスト・高速）を使用する場面:
  - コードフォーマット・lint 自動修正
  - 単純なファイルリネーム・移動
  - CHANGELOG / コミットメッセージ生成
  - 単純な型エラー修正
  - テンプレートファイルの生成

claude-sonnet（バランス型・標準使用）を使用する場面:
  - 新機能の実装（中規模）
  - バグ修正（原因が特定済み）
  - コードレビュー・テスト生成
  - ドキュメント生成
  - リファクタリング（スコープ明確）

claude-opus（高性能・複雑タスク）を使用する場面:
  - アーキテクチャ設計・技術選定
  - 複雑なバグ（根本原因不明）
  - セキュリティ脆弱性の深堀り分析
  - 大規模リファクタリングの計画立案
  - エスカレーション判断が必要な状況
```

### コンテキスト管理ルール

```
コンテキスト使用量の最適化:
  1. /compact を使用するタイミング:
     - コンテキスト使用率が 70% を超えたとき
     - タスクが完了して次のタスクに移るとき
     - セッションが2時間を超えたとき

  2. ファイル読み込みの優先順位（必要最小限に絞る）:
     優先度 HIGH: 直接変更するファイル
     優先度 MED:  変更ファイルが import するファイル
     優先度 LOW:  型定義・設定ファイル
     除外:        node_modules / dist / .git / build 成果物

  3. 長い出力の圧縮:
     - テスト結果: 失敗件数と失敗メッセージのみ（通過テストは省略）
     - diff 表示: 変更行のみ（前後3行のコンテキスト）
     - ログ出力: エラー行のみ抽出
```

### バッチ処理ルール

```
以下の操作は個別ではなくバッチで実行する:
  - 複数ファイルの lint 自動修正 → 一括実行
  - import 文の整理 → 全ファイル一括
  - 型定義の更新 → 関連ファイル一括
  - コメントの追加 → 同一モジュール内一括

バッチサイズ上限:
  - 1バッチあたり最大 20 ファイル
  - 超える場合は関連性の高いグループに分割
```

## エラーハンドリングルール

### エラー種別と対応アクション

| エラー種別 | 自動対応 | リトライ | エスカレーション閾値 |
|-----------|---------|---------|------------------|
| コンパイルエラー | ✅ 自動修正 | 最大3回 | 3回失敗 |
| テスト失敗（原因明確） | ✅ 自動修正 | 最大3回 | 3回失敗 |
| テスト失敗（原因不明） | ⚠️ 分析後判断 | 最大1回 | 1回失敗 |
| lint エラー | ✅ 自動修正 | 最大2回 | 2回失敗 |
| セキュリティ警告 | 🛑 即時停止 | なし | 即座に |
| ネットワークエラー | ✅ リトライ | 最大5回 | 5回失敗 |
| タイムアウト | ⚠️ 処理分割 | 最大2回 | 2回失敗 |
| 権限エラー | 🛑 即時停止 | なし | 即座に |

### カスケード障害の検出と対応

```
カスケード障害の兆候:
  - 修正 A が問題 B を引き起こし、修正 B が問題 A を引き起こす
  - エラー数が修正を重ねるたびに増加している
  - 同一ファイルを3回以上修正している

カスケード障害検出時の対応:
  1. git stash で全変更を退避
  2. 問題の依存関係ツリーを分析
  3. 最も根本的な問題を特定
  4. 影響範囲の小さい変更から再開
  5. 解決できない場合はエスカレーション
```

### 外部依存エラーの処理

```
GitHub API / CI エラーの場合:
  - 最大5回リトライ（指数バックオフ: 1s, 2s, 4s, 8s, 16s）
  - 5回失敗でオフラインモードに切り替え
  - オフラインモードではローカル操作のみ継続

外部サービス（Stripe / AWS / Firebase 等）エラーの場合:
  - モックに切り替えてテスト継続
  - 実サービスへの接続エラーは即座に人間に通知
  - 絶対にリトライで本番データを操作しない
```

## プロジェクトオンボーディングプロセス

新しいプロジェクトで初めて起動したとき、以下を実行します:

```
Phase 1: リポジトリ分析（最初の5分）
  1. ls / find を使ってプロジェクト構造を把握
  2. package.json / Cargo.toml / go.mod 等でスタックを確認
  3. README.md / AGENTS.md を読む
  4. git log --oneline -20 で最近の作業を把握
  5. 開発環境の確認（.env.example の存在確認）

Phase 2: 品質ベースライン測定
  1. 既存テストを実行（失敗数を記録）
  2. lint を実行（既存警告数を記録）
  3. coverage を計測（現在のベースラインを記録）
  → これらは「フリーズライン」として記録（悪化させない基準）

Phase 3: TASKS.md が存在しない場合
  1. GitHub Issues を確認
  2. オープン Issues を優先度順に TASKS.md に変換
  3. AGENTS.md が存在しない場合はプロジェクト分析から生成を提案

Phase 4: 作業開始宣言
  「セッション開始プロトコル」のフォーマットで宣言してから実装開始
```

## 禁止事項（グローバル適用）

以下はすべてのプロジェクトで絶対に行ってはなりません:

```
NEVER DO:
  ❌ シークレット・APIキー・パスワードをコードにハードコード
  ❌ main / master への直接プッシュ
  ❌ テスト失敗状態でのコミット
  ❌ // @ts-ignore の理由なし使用
  ❌ 本番 DB への直接接続・操作
  ❌ eval() / exec() / Function() の使用（テストコードも含む）
  ❌ console.log のリリースコードへの混入（テストは除く）
  ❌ 認証なし新規公開エンドポイントの追加
  ❌ 依存関係の理由なしアップグレード（メジャーバージョン）
  ❌ エスカレーション条件での自律的判断・続行
  ❌ .env ファイルの Git 追加
  ❌ node_modules / dist / build 成果物の Git 追加
  ❌ 既存テストを「通過させるために」弱体化・削除
```
