# 04 リファクタリング（Refactoring）

## 概要

技術的負債の解消・可読性向上・テスト容易性の改善を、
既存テストを維持しながら段階的に実施します。

---

## Claude Code 起動コマンド

```bash
claude --dangerously-skip-permissions
```

---

## プロンプト指示

```
以下のコードをリファクタリングしてください。

【対象】
- ファイル / モジュール: [FILE_PATH or MODULE_NAME]
- 現在の問題点（該当するものにチェック）:
  - [ ] コードが長すぎる（関数が100行超など）
  - [ ] 重複コードが多い（DRY 原則違反）
  - [ ] 命名が不明確
  - [ ] テストが書きにくい構造（密結合）
  - [ ] 型が不十分（any だらけ）
  - [ ] 循環的複雑度が高い（分岐が多すぎる）
  - [ ] その他: [詳細]

【方針（Architect / 該当 Dev で議論）】
- 外部インターフェース（API・関数シグネチャ）は変更しない
- 既存テストをすべてパスさせる（回帰テストとして活用）
- リファクタリング後に新規テストを追加する
- パフォーマンスを悪化させない

【実行してほしいこと】
1. 現状コードを分析して問題点・改善効果を整理する
2. リファクタリング計画（フェーズ分け）を提示して承認を得る
3. フェーズ1: テストカバレッジが不足していれば先にテストを追加する
4. フェーズ2: 段階的にリファクタリングを実施する
   - 1ステップごとにテストを実行して回帰がないことを確認する
   - 各ステップを個別にコミットする（git bisect しやすくする）
5. 変更の意図をコメントに記載する（なぜこの設計にしたか）
6. リファクタリング後にメトリクスを比較する
   - 循環的複雑度（before / after）
   - テストカバレッジ（before / after）
7. git commit（refactor: [対象モジュール]）する

【完了基準】
- 全テストがパス
- lint / typecheck がパス
- 循環的複雑度が改善されている
- コードの可読性が向上している（レビュー者の主観評価含む）
```

---

## 使用場面

| シナリオ | 説明 |
|---------|------|
| 技術的負債の解消 | コードの複雑度・保守性の改善 |
| 大規模機能追加の前準備 | 安全に変更できるコード基盤を作る |
| オンボーディング前 | 新メンバーが読みやすいコードに整理 |
| テスト追加の前準備 | テストしやすい構造へのリファクタリング |

---

## リファクタリングパターン

| パターン | 適用場面 |
|---------|---------|
| Extract Function | 長い関数の分割（20行を目安に） |
| Extract Class | 複数の責務を持つクラスの分割 |
| Replace Conditional with Polymorphism | 巨大な if/switch の整理 |
| Introduce Parameter Object | パラメータが多い関数のグループ化 |
| Replace Magic Number | マジックナンバーを定数に置き換え |
| Dependency Injection | テスト容易性のための依存関係の外部化 |

---

## メトリクス計測コマンド

```bash
# TypeScript: 複雑度チェック（eslint complexity）
npx eslint src/ --rule '{"complexity": ["error", 10]}' 2>&1

# Python: radon で循環的複雑度を計測
pip install radon && radon cc src/ -a

# テストカバレッジ
npm test -- --coverage
```

---

## ポイント

- 一度に大きく変えず、段階的リファクタリングを指示するとリスクが下がる
- 「外部インターフェースを変更しない」制約が誤った破壊的変更を防ぐ
- 各ステップを個別コミットにすることで問題発生時の切り戻しが容易
- Triple Loop の Build Loop 内でインクリメンタルにリファクタリングするのも効果的
