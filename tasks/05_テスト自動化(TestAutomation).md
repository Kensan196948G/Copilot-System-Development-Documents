# 05 テスト自動化（Test Automation）

## 概要

対象コードのテスト戦略を設計し、ユニット・統合・E2E テストを自動生成。
カバレッジ 80%以上を目標に整備します。

---

## Claude Code 起動コマンド

```bash
claude --dangerously-skip-permissions
```

---

## プロンプト指示

```
以下のモジュール / 機能に対してテストを自動生成・整備してください。

【対象】
- テスト対象: [FILE_PATH or MODULE_NAME]
- テストフレームワーク: [Jest / Vitest / pytest / Go test など]
- 現在のカバレッジ: [xx% または「未計測」]

【テスト種別（Tester Agent が担当）】
- [ ] ユニットテスト（純粋関数・クラスメソッド）
- [ ] 統合テスト（API エンドポイント・DB 操作）
- [ ] E2E テスト（Playwright / Cypress）
- [ ] スナップショットテスト（UI コンポーネント）
- [ ] パフォーマンステスト（負荷テスト・ベンチマーク）

【実行してほしいこと】
1. 対象コードを解析してテスト戦略を提示する
   - どの種別のテストが何個必要か
   - モック・スタブが必要な依存関係の洗い出し
2. 境界値・異常系・ハッピーパスをすべてカバーするテストケースを設計する
3. テストコードを生成する
   - テストの独立性を確保する（beforeEach でリセット）
   - モックは最小限にする（過剰なモックはテストの信頼性を下げる）
4. テストを実行してすべてパスすることを確認する
5. カバレッジレポートを生成して改善前後を比較する
6. カバレッジが 80% 未満の場合は追加テストを生成する
7. git commit（test: [対象モジュール] のテストを追加）する

【品質基準】
- ライン / ブランチカバレッジ 80% 以上
- テストは独立して実行できる（副作用なし）
- テスト実行時間が適切（ユニット: < 1秒/テスト、E2E: < 30秒/テスト）
```

---

## 使用場面

| シナリオ | 説明 |
|---------|------|
| CI に自動テストを追加 | 新規・既存コードのテストカバレッジ向上 |
| リファクタリング前の安全網 | 変更前に動作保証のテストを用意 |
| TDD（テストファースト） | テストを先に書いて実装を導く |
| 回帰テストの整備 | バグ修正後の再発防止テスト |

---

## テストフレームワーク別コマンド

```bash
# Jest（Node.js / TypeScript）
npx jest --coverage --verbose

# Vitest（Vite プロジェクト）
npx vitest run --coverage

# pytest（Python）
pytest --cov=src --cov-report=html -v

# Go test
go test ./... -cover -v

# Playwright（E2E）
npx playwright test --reporter=html
```

---

## テストケース設計テンプレート

```
【関数名】getUserById(id: string)

テストケース:
  正常系:
  - 有効な ID で正しいユーザーを返す
  - 管理者ユーザーの正しい権限を返す

  異常系:
  - 存在しない ID で null を返す
  - 無効な形式の ID で ValidationError をスローする

  境界値:
  - ID が空文字列の場合
  - ID が最大長の場合
```

---

## ポイント

- カバレッジ目標を明示することで必要十分なテストが生成される
- 「モックは最小限」の制約が過剰モックによる偽陽性テストを防ぐ
- Triple Loop の Verify Loop でテスト実行・カバレッジ確認が自動化される
- テスト追加は小さな PR に分けると CI がわかりやすくなる
