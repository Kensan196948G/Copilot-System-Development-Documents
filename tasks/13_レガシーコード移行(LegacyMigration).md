# 13 レガシーコード移行（Legacy Code Migration）

## 概要

Strangler Fig パターンを採用した段階的移行で、既存機能を維持しながら
現代的な技術スタックへ安全に移行します。

---

## Claude Code 起動コマンド

```bash
claude --dangerously-skip-permissions
```

---

## プロンプト指示

```
以下のレガシーコードを現代的な技術スタックに移行してください。

【現状】
- 現在の技術: [JavaScript / Python 2 / jQuery / PHP 5 / CJS など]
- 移行先の技術: [TypeScript / Python 3.12 / React / PHP 8 / ESM など]
- 対象ファイル / モジュール: [PATH or 全体]
- コードベースの規模: [ファイル数 xx / 行数 xx 万行]
- 現在のテストカバレッジ: [xx% または「テストなし」]

【移行方針（Architect / CTO Agent が設計）】
- Strangler Fig パターン（段階的移行）を採用する
- 既存機能を完全に維持する（機能追加は移行後）
- テストファースト移行（先にテストを書いてから移行する）
- パフォーマンスを悪化させない
- ロールバック手順を事前に準備する

【実行してほしいこと】
1. コードベースを解析して移行の難易度・リスクを評価する
   - 依存関係グラフを作成する
   - 移行困難な箇所（外部 API / DB スキーマ / 型なし動的コード）を洗い出す
2. 移行計画（フェーズ分け・優先度・期間）を提示して承認を得る
3. フェーズ0: 現状のコードに対するテストを追加する
   （移行前の動作を保証するカラーテスト）
4. フェーズ1: 段階的に新技術へ移行する
   - 依存関係の少ない末端のモジュールから開始する
   - 各モジュール移行後にテストを実行して回帰がないことを確認する
5. フェーズ2: 旧コードを削除する
6. 移行完了後に型安全性・lint・テストカバレッジをチェックする
7. 各フェーズを個別の PR にして段階的にマージする
8. git commit（migrate: [対象] を [新技術] に移行）する

【リスク管理】
- 移行中は旧コードと新コードを並行して動かせる設計にする
- feature flag で新旧コードを切り替えられるようにする
- ロールバック手順を README に記載する
```

---

## 使用場面

| シナリオ | 説明 |
|---------|------|
| JavaScript → TypeScript 移行 | 型安全性の導入 |
| CommonJS → ESModules 移行 | モジュールシステムの現代化 |
| レガシー API の REST 化 | SOAP / RPC → REST / GraphQL |
| Python 2 → Python 3 移行 | バージョンアップ |
| jQuery → React/Vue 移行 | フレームワーク移行 |

---

## 移行の進め方（Strangler Fig パターン）

```
フェーズ0: 現状把握・テスト追加
  ↓ テストカバレッジ確保
フェーズ1: 末端モジュールから移行
  module_a.js → module_a.ts ✅
  module_b.js → module_b.ts ✅
  ↓ 依存グラフに沿って順番に
フェーズ2: コアモジュールの移行
  core.js → core.ts ✅
  ↓
フェーズ3: エントリポイントの移行
  index.js → index.ts ✅
  ↓
フェーズ4: 旧コードの削除・クリーンアップ
```

---

## JavaScript → TypeScript 移行チェックリスト

```bash
# TypeScript の段階的導入
# tsconfig.json に allowJs: true を設定して .js と .ts を混在させる

# any の使用状況を確認
grep -r ": any" src/ | wc -l

# strict モードの段階的有効化
# 1. "strict": false → テストがパスすることを確認
# 2. "strictNullChecks": true → null チェックを追加
# 3. "strict": true → 全オプションを有効化
```

---

## ポイント

- 段階的移行を強調することで一気に変えるリスクを避けられる
- 「テストファースト移行」で機能の退行を確実に検知できる
- feature flag があると本番で新旧コードを安全に切り替えられる
- 移行期間中は旧コードへのバグ修正を最小限にする（コンフリクト防止）
