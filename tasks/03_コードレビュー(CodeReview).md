# 03 コードレビュー（Code Review）

## 概要

PR・ファイル・差分を多角的な観点でレビューし、重大度別に指摘事項を整理します。
Minor 以下は自動修正まで実施します。

---

## Claude Code 起動コマンド

```bash
claude --dangerously-skip-permissions
```

---

## プロンプト指示

```
以下のコードまたはプルリクエストをレビューしてください。

【レビュー対象】
- ファイル / PR: [FILE_PATH or PR_URL or ブランチ名]
- 変更内容の概要: [どんな変更か]
- 変更の背景・目的: [なぜこの変更が必要か]

【レビュー観点（Agent Teams: QA / Architect / Security）】
1. 🔍 正確性・ロジック
   - バグ・エッジケースの見落とし
   - 型安全性・null 安全性
   - 非同期処理の正確性（race condition / deadlock）
2. 🏗 設計・保守性
   - SOLID 原則への準拠
   - 関数・クラスの責務の明確さ（単一責任）
   - 命名の適切さ・可読性
   - 重複コード（DRY 原則）
3. ⚡ パフォーマンス
   - 不必要なループ・再計算
   - N+1 問題・メモリリーク
   - 不適切なキャッシュ戦略
4. 🔐 セキュリティ
   - インジェクション・XSS・CSRF リスク
   - 機密情報のハードコード
   - 認証・認可の抜け漏れ
5. 🧪 テスト
   - テストカバレッジの十分さ
   - テストケースの網羅性（正常系・異常系・境界値）
   - テストの独立性・再現性

【出力形式】
- 重大度: 🔴 Critical / 🟠 Major / 🟡 Minor / 🟢 Suggestion
- 各指摘に対して改善コードを提示する
- 問題がなければ LGTM と明記する

レビュー完了後、自動修正可能な Minor 以下の指摘は修正してコミットしてください。
Critical / Major はユーザーに確認してから修正します。
```

---

## 使用場面

| シナリオ | 説明 |
|---------|------|
| PR マージ前の品質チェック | GitHub PR URL を渡すだけで自動レビュー |
| セルフレビュー | 自分のコードの客観的な評価 |
| セキュリティ観点のみ | `【レビュー観点】` で 🔐 のみを指定 |
| 設計レビュー | アーキテクチャ・インターフェース設計の確認 |

---

## 差分の渡し方

```bash
# PR 全体を渡す（推奨）
gh pr diff [PR番号] | head -500

# 特定ファイルの差分
git diff main -- src/api/users.ts

# ブランチ間の差分
git diff main...feature/new-endpoint --stat

# 最新コミットの差分
git show HEAD
```

---

## レビューの重大度基準

| 重大度 | 基準 | 対応 |
|-------|------|------|
| 🔴 Critical | バグ・セキュリティ脆弱性・データロスリスク | マージ前に必ず修正 |
| 🟠 Major | 設計上の問題・パフォーマンス劣化 | 修正推奨（議論要） |
| 🟡 Minor | コードスタイル・命名・可読性 | 自動修正してコミット |
| 🟢 Suggestion | より良い書き方の提案 | 任意対応 |

---

## ポイント

- `git diff main` の出力を貼り付けると精度が上がる
- 「自動修正可能な指摘は修正してコミット」を入れることで効率化できる
- `変更の背景・目的` を伝えることでレビュー精度が向上する
- Triple Loop の Verify Loop でも同様のレビューが自動実施される
