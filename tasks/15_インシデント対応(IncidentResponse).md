# 15 インシデント対応（Incident Response）

## 概要

本番インシデント発生時の緊急対応フロー。ログ調査・暫定対処・根本修正・
ポストモーテムまでを構造化した手順で実施します。

---

## Claude Code 起動コマンド

```bash
claude --dangerously-skip-permissions
```

---

## プロンプト指示

```
本番環境でインシデントが発生しています。緊急対応してください。

【インシデント概要】
- 発生時刻: [YYYY-MM-DD HH:MM JST]
- 影響範囲: [全ユーザー / 一部ユーザー xx% / 特定機能]
- 症状:
  - [ ] サービス全停止（500 エラー多発）
  - [ ] 特定機能が動作しない
  - [ ] パフォーマンス劣化（レスポンスタイム xx ms → xx ms）
  - [ ] データ不整合
  - [ ] セキュリティインシデント（不正アクセス等）
- エラーログ:
  ```
  [エラーログをここに貼り付け]
  ```
- 直前の変更（あれば）: [デプロイ / 設定変更 / トラフィック急増]

【緊急対応手順（CTO / Ops / 該当 Dev Agent が担当）】
1. 【即時】ログ・メトリクスを確認して根本原因を特定する
   - アプリケーションログ・インフラメトリクス・DB スロークエリ
2. 【即時】影響を最小化する暫定対処
   - フィーチャーフラグ OFF
   - 旧バージョンへのロールバック（git revert）
   - トラフィックの切り離し
3. 【短期】根本修正を実装してテストする
4. 【確認後】本番環境へ適用する
   ⚠️ git push（本番適用）は必ずユーザー確認後に実施する
5. 【事後】インシデントレポートを生成する
   docs/incidents/YYYY-MM-DD_[概要].md

【エスカレーション基準】
- 30 分で解決しない場合 → エスカレーション候補者に通知
- データロスの可能性がある場合 → 即時エスカレーション
- セキュリティインシデントの場合 → 即時エスカレーション + 外部機関への報告検討

完了後にポストモーテムドキュメントを作成してください。
```

---

## 使用場面

| シナリオ | 説明 |
|---------|------|
| 本番障害の緊急対応 | サービス停止・機能不全の復旧 |
| パフォーマンス劣化の原因調査 | SLA 違反の対応 |
| データ不整合の修正 | DB 整合性の復旧 |
| セキュリティインシデント | 不正アクセス・情報漏洩の対応 |

---

## インシデント対応タイムライン

```
T+0     インシデント検知（アラート / ユーザー報告）
T+5分   影響範囲・症状の確認
T+10分  暫定対処の実施（ロールバック等）
T+30分  根本原因の特定
T+60分  根本修正の実装・テスト
T+90分  本番適用（ユーザー確認後）
T+24h   ポストモーテム作成
T+7日   再発防止策の実装完了
```

---

## ポストモーテムテンプレート

```markdown
# インシデントレポート: [YYYY-MM-DD] [概要]

## 概要
- 発生時刻: YYYY-MM-DD HH:MM JST
- 復旧時刻: YYYY-MM-DD HH:MM JST
- 影響時間: xx 分
- 影響ユーザー数: xx 人（全体の xx%）

## 根本原因

## タイムライン

| 時刻 | 対応内容 | 担当 |
|-----|---------|-----|
| HH:MM | インシデント検知 | - |
| HH:MM | 暫定対処実施 | [名前] |
| HH:MM | 根本修正デプロイ | [名前] |

## 再発防止策

- [ ] [対策1] （期限: YYYY-MM-DD）
- [ ] [対策2]

## 良かった点 / 改善点
```

## ロールバック手順

```bash
# 直前のコミットに戻す
git revert HEAD
git push origin main

# 特定のコミットに戻す
git revert [commit-sha]

# タグで管理している場合
git checkout v1.2.3
# → Vercel / Railway / AWS ECS で前バージョンのイメージを指定
```

---

## ポイント

- **git push（本番適用）は必ずユーザーが確認してから行う**
- ポストモーテムを残すことで再発防止に繋げる（blame より改善にフォーカス）
- P0 インシデントは Slack / PagerDuty での即時通知が前提
- Triple Loop の Monitor Loop がインシデントの早期検知に貢献する
