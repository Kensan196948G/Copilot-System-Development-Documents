# コスト最適化ガイド — プレミアムリクエスト管理

> **目的**: GitHub Copilot CLI のプレミアムリクエスト消費量を把握し、Triple Loop 15H を持続的に運用するためのコスト最適化戦略を提供します。

---

## 1. プレミアムリクエストとは

GitHub Copilot CLI でモデルにリクエストするたびに「プレミアムリクエスト」が消費されます。月次上限に達すると、その月の残りはサービスが制限されます。

### モデル別コスト倍率（目安）

| モデル | 倍率 | 推奨用途 |
|-------|------|---------|
| Claude Opus 4.6 | 高（10×） | 設計・複雑なロジック（最小限に） |
| Claude Opus 4.5 | 高（10×） | 上記の旧版 |
| Claude Sonnet 4.6 | 中（1×）※基準 | 一般的な実装（デフォルト） |
| GPT-5.3-Codex | 中（2×） | コード生成・変換 |
| GPT-5.2 | 中（1.5×） | バランス型 |
| Claude Haiku 4.5 | 低（0.25×） | 単純処理・サブエージェント |
| GPT-5-mini | 低（0.1×） | 軽量タスク |

> **確認方法**: `/model` コマンドで現在のモデルと倍率を確認できます。

## 2. Triple Loop 15H の消費量シミュレーション

### シングルエージェント（通常モード）

| フェーズ | 平均リクエスト数 | モデル |
|---------|--------------|-------|
| Monitor Loop（Issue分析・TASKS.md更新） | 5〜10 | Sonnet 4.6 |
| Build Loop（1タスク実装） | 15〜30 | Sonnet 4.6 |
| Verify Loop（レビュー・テスト） | 5〜10 | Sonnet 4.6 |
| **1タスクあたり合計** | **25〜50** | |

**15時間で処理できるタスク数の目安**（Sonnetデフォルト）:

| 月次上限（リクエスト数） | 1タスク=30リクエストとして |
|---------------------|------------------------|
| 300 | 〜10タスク |
| 1,000 | 〜33タスク |
| 3,000 | 〜100タスク |

### /fleet 使用時（並列エージェント）

```
/fleet でサブエージェント N 個起動した場合:
消費量 ≒ シングルの 1.5〜2倍（並列処理でコンテキストが分散）

例: 3サブエージェント並列
  - backend-agent (Opus 4.6 × 10倍): 15リクエスト × 10 = 150相当
  - test-writer (GPT-5.3-Codex × 2倍): 10リクエスト × 2 = 20相当
  - docs-agent (Sonnet 4.6 × 1倍): 5リクエスト × 1 = 5相当
  合計: 175相当リクエスト（1タスク）
```

## 3. コスト最適化パターン

### パターン1: モデル使い分け戦略（最重要）

```
❌ 非効率（全タスクにOpus使用）
/fleet 全タスクを Claude Opus 4.6 で実装してください

✅ 効率的（タスク別モデル指定）
/fleet
  Use Claude Opus 4.6 でシステム設計・複雑なアルゴリズムのみ実装し、
  Use Claude Sonnet 4.6 で通常のAPIコード実装を行い、
  Use Claude Haiku 4.5 でコメント追加・ドキュメント整備を実施してください
```

**節約効果目安**: 同じタスクで 60〜80% のリクエスト削減

### パターン2: サブエージェントへの安価モデル割り当て

`.github/agents/` のカスタムエージェントでモデルを明示する:

```yaml
# test-writer.agent.md — 単純なテスト生成はHaikuで十分
model: claude-haiku-4.5  # 0.25倍コスト

# docs-agent.agent.md — ドキュメント生成もSonnetで十分
model: claude-sonnet-4.6  # 1倍コスト

# backend-agent.agent.md — 複雑なロジックはOpusを使う
model: claude-opus-4.6  # 10倍コスト ← 使用を最小限に
```

### パターン3: /compact による定期コンテキスト圧縮

コンテキストが長くなるほど各リクエストのトークン数が増加:

```
推奨タイミング:
  - 各タスク完了後
  - コンテキスト使用率 70% 超えたら
  - セッション開始から 2時間経過したら

実行:
  /compact  # 会話履歴を自動圧縮
```

**節約効果目安**: コンテキスト圧縮で 20〜40% のトークン削減

### パターン4: 事前調査の効率化

```
❌ 非効率（毎回コードを読んで考える）
「このプロジェクトのAPIを実装してください」

✅ 効率的（コンテキストを最初に注入）
「@src/api/users/get-user.ts と @AGENTS.md を参照して、
 同じパターンで PATCH /api/v1/users/:id/profile を実装してください」
```

**節約効果目安**: ファイル参照で探索コストを 30〜50% 削減

### パターン5: タスク分割サイズの最適化

```
/fleet の最適サブタスク数: 2〜4個

多すぎる例（8サブエージェント → コスト2倍以上）:
  /fleet 以下を並列で実装: API・テスト・ドキュメント・Dockerfile・
         CI設定・監視設定・ログ設定・セキュリティ設定

最適な例（3サブエージェント）:
  /fleet
    Use @backend-agent でAPI・Dockerfile・CI設定を実装し、
    Use @test-writer でテストを生成し、
    Use @docs-agent でドキュメントを更新してください
```

## 5. プラン別推奨運用戦略

| プラン目安 | 推奨戦略 |
|-----------|---------|
| 〜300リクエスト/月 | Opus禁止・Haiku多用・/fleet使用禁止 |
| 300〜1,000/月 | Sonnetをデフォルト・Opusは設計時のみ |
| 1,000〜3,000/月 | パターン1〜4を適用・週2回Triple Loop |
| 3,000+/月 | フルに/fleet活用・毎日Triple Loop可能 |

## 6. コスト削減チェックリスト

実施前に確認:

```
□ タスクに使うモデルを明示したか？（Opus乱用を避ける）
□ @ファイルパス でコンテキストを注入したか？
□ /fleet のサブエージェント数は4個以内か？
□ カスタムエージェントにモデルを指定したか？
□ 2時間ごとに /compact を実行したか？
□ 探索タスクは explore エージェントを使ったか？
□ 単純タスクに Haiku/GPT-5-mini を使ったか？
```
